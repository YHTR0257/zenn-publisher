**Key Points**
- 自作3構成（AMD+Nvidia / AMD+AMD / Intel+Nvidia）はいずれも¥270,000〜¥280,000で価格差がほぼなく、帯域幅と電力効率のトレードオフで選択が分かれる
- 帯域幅最大はAMD + RX 9070 XT（640 GB/s）。DGX Spark（$3,000超）を超えるコスパ

- 秘書タスク用途ではHaiku+Sonnetミックス（月$44≈¥6,600）でローカルの回収期間は約3.5年。まずAPIで始めて使用量を把握してからローカル移行を検討するのが現実的
- クラウドの使い放題プランが持続しない可能性があり、従量課金化が進むとローカルの損益分岐点はさらに前倒しになる

# はじめに
色々な人や組織がエージェントについての実装例を上げ始めています。サーバー上のワークフローからOpenClawを始めとしたローカル操作まで、色々な事例があります。
その流れの中で自分もLLMで専属秘書が欲しい！！！と強く思い始めました。
選択肢はAPI/Cloud Service/Localの3種類です。とはいえ、無限の資産があるわけではありません・・・

そこでいろいろな実行基盤を比較して自分に一番適したものは何かを探していこうと思います。

# 前提条件
まずは何に使用したいかを定義しておきます。秘書として最低限の役割で考えてみます。自動での予約などはワークフローの整理が難しいので後ですかね・・・

:::message
**使用用途**
-  Notionのタスクページを参照して、実行日に割り当てられているタスクをSlack/LINEなどに通知する(通知文を生成することも担当)
- メールの中から重要なドキュメントを特定し、特定のストレージに保存する
:::

ここから必要となりそうなモデルについて考えてみます。

:::message
**選び方**
- モデルサイズは20GB(個人利用・単発タスクのため)
- [LLM Stats](https://llm-stats.com/)のTool CallingとWritingのボードを参考
- 必要サイズはモデルサイズの1.2倍としています。
:::

| モデル名                                       | 説明                                                                | 必要サイズ               |
| ------------------------------------------ | ----------------------------------------------------------------- | ------------------- |
| Qwen/Qwen3.6-35B-A3B                       | 安心安全のQwenシリーズ。軽量さと性能を両立。MoEを使用。                                   | 23GB(UD-IQ4_NL)     |
| google/gemma-4-26B-A4B-it                  | Gemmaシリーズの軽量モデル。MoEを使用した。                                         | 13.6 GB(UD-IQ4_NL)  |
| Qwen/Qwen3.6-27B                           | QwenシリーズのDenseモデル。MoEなしで全27Bパラメータが推論に使用されるため品質が安定。             | 23.1GB(UD-IQ4_NL)   |
| google/gemma-4-31B-it                      | Gemmaシリーズのマルチモーダルモデル。 **最大256Kトークンの長文コンテキスト** を扱え、**140以上の言語**を学習 | 21.75GB (UD-IQ4_NL) |
| nvidia/NVIDIA-Nemotron-3-Nano-30B-A3B-BF16 | NVIDIAのMoEモデル。30Bパラメータ中3Bが推論に使用。Tool CallingとStructured Outputに強みを持つ。 | 18.5GB (Q4_K_M)     |

これらを踏まえて必要となるメモリ量は以下のようになります。
- DRAM: 32GB
	- 関連サービスを運用するサイズ。
- VRAM: 16~24GB
	- モデルを乗せ切ることが出来るサイズ。

# 検討

検討するのは以下の5種類です。

| 構成 | CPU | GPU |
| -- | --- | --- |
| AMD + Nvidia | Ryzen 7 9600X | RTX 5060 Ti |
| AMD + AMD | Ryzen 7 9700X | RX 9070 XT |
| Intel + Nvidia | Core Ultra 7 265K | RTX 5060 Ti |
| Mac Mini | Apple M4 Pro | 統合GPU |
| AMD Ryzen™ AI Max+ 395 | Ryzen AI Max+ 395 | 統合GPU（40CU） |

検討事項としては
- 導入コスト
	- 機器購入費用
- 運用コスト
	- 電気代
- データの転送速度 (帯域幅)
	- 応答速度に影響

# 各構成の特徴

帯域幅はLLM推論速度に直結します。参考として、$3,000超えの DGX Spark（GB10）の統合メモリ帯域幅は273 GB/sです。

| 構成 | メモリ帯域幅 |
| -- | --------- |
| AMD + RX 9070 XT | 640 GB/s |
| AMD/Intel + RTX 5060 Ti | 448 GB/s |
| Mac Mini (M4 Pro 48GB) | 273 GB/s |
| AMD Ryzen™ AI Max+ 395 | 256 GB/s |

✅メリット

- VRAM 16GBに収まらないモデルはCPU推論にフォールバックする
- モデルの大型化が進むと上限に当たりやすい

✅メリット
- **5構成中最大の帯域幅（640 GB/s）** → トークン生成速度で最速の見込み

# まとめ

各指標は5点満点（高いほど良い）で評価します。

| 評価指標 | 5点の基準 | 1点の基準 |
| ------ | ------- | ------- |
| 導入コスト | 安い（〜24万円） | 高い（35万円〜） |
| 運用コスト | 低消費電力（〜30W） | 高消費電力（220W〜） |
| 帯域幅 | 高い（640 GB/s） | 低い（256 GB/s） |

| 構成 | 概算価格 | 導入コスト | 運用コスト | 帯域幅 | 合計 |
| -- | ---: | :------: | :------: | :---: | :--: |
| Mac Mini（48GB） | ¥248,800 | 5 | 5 | 2 | **12** |
| AMD + Nvidia（RTX 5060 Ti） | ¥271,480 | 3 | 3 | 4 | **10** |
| AMD + AMD（RX 9070 XT） | ¥271,480 | 3 | 2 | 5 | **10** |
| Intel + Nvidia（RTX 5060 Ti） | ¥279,599 | 3 | 3 | 4 | **10** |
| AMD Ryzen™ AI Max+ 395 | ¥350,000〜 | 1 | 4 | 1 | **6** |

実際に価格を計算すると、**Mac Mini（48GB）が¥248,800で最安**となりました。自作3構成はいずれも¥270,000前後に収まり、コスト差はほとんどありません。AMD + Nvidia と AMD + AMD は同額（¥271,480）で、帯域幅と運用コストのトレードオフで選択が分かれます。自作3構成の価格差は最大¥8,000程度で実質横並びです。導入コストのスコアはいずれも3点となりました。

帯域幅を最優先にするなら **AMD + RX 9070 XT**、セットアップの手軽さと電力効率を重視するなら **Mac Mini** が有力候補です。

## ちょっと待って、Cloud でよくない？

ここまで読んで気づいた人もいると思うんですけど、**どの構成も25〜28万円前後**なんですよね。それだけ払うなら、Claude APIとかOpenAI APIを月いくら使えるの？って話になってきます。

### 実際の使用量で計算してみた

筆者の月間使用量を2ヶ月分で見てみると：

| 月 | 使用量 | 内訳 |
| -- | -----: | ---- |
| 3月 | 35,196,681 tokens | ほぼコーディング |
| 4月 | 177,076,350 tokens | コーディング70%、その他30% |

3月のほぼ全量がコードであることを踏まえると、**ヘビーユーザーの月間使用量は50M tokens前後**がひとつのラインになりそうです。ただし秘書系タスクに絞ると話は変わってきます。APIツール連携はnginxでホストして別実装にすれば、LLMに投げるリクエスト自体はそこまで多くありません。通知文の生成やメール要約程度であれば、**約20M tokens/月**で十分収まりそうです。

モデルの使い分けとしては、単純タスク（通知文生成など）は **Claude Haiku や GPT-5.5** 、メール解析・判断が必要な複雑タスクは **Sonnet クラス**が現実的なターゲットになります。

Haiku 70% / Sonnet 30% のミックスで計算すると：

| モデル | 割合 | トークン数 | 単価 | 費用 |
| ---- | --: | -------: | ---: | ---: |
| Claude Haiku | 70% | 14M tokens | $1/1M | $14 |
| Claude Sonnet | 30% | 6M tokens | $5/1M | $30 |
| **合計** | | **20M tokens** | | **$44/月（≈¥6,600）** |

AMD構成（¥271,480）の回収期間は **約41ヶ月（3.5年）** となりました。ちょっと迷うぐらいの金額ですね。ローカルだと使い放題になるため心理的安全性は高くなります。自分のトークン使用量やモデル選定次第でコスパが大きく変わるのが正直なところです。

そしてこの数字を見てもうひとつ気づくことがあります。それは筆者のような使い方をされたら、サービス側としては**使い放題プランが持続しなさそう**なんですよね。月40〜50M tokensを定額で提供し続けるのはどう考えても赤字です。これからは従量課金制に移行していくのでしょうか？そうなると「CloudかLocalか」の損益分岐点はさらに前倒しになってきます。
こうなると、
- **「とりあえずAPIで始めてみて、使用量が増えてきたらローカルに移行する」** 
- 「高い知能や処理性能を必要とする場合はAPIを、単純なタスクや文章要約にはローカルLLMを使用する」
などの段階的/階層的なアプローチが現実的かもしれません。


:::message
**それでもローカルが有利なのはこんなとき**
- **プライバシー最優先**：データを外に出したくない
- **オフライン環境**：ネットがない or 不安定な環境で動かしたい
- **コストの固定化**：ランニングコストをゼロにしたい（電気代は除く）
- **カスタマイズの自由度**：ファインチューニングや特殊な推論設定をしたい
- **ヘビーユーザー**：月間トークン使用量が多く、API費用がかさんできた
:::

「とりあえずAI秘書として動かしたい」「まず試してみたい」という段階なら、**APIから始めるのが圧倒的に早くて安い**です。まず使用量を把握してから、ローカル移行を検討するのが現実的なステップかもしれません。
